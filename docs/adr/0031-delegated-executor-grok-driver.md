# ADR-0031 · Delegated executor: rote tasks driven through the grok CLI by a verifying Claude driver

**Context.** ADR-0030's `via: grok-cli` needs an execution protocol. The motive is budget: rote work off the Claude usage budget, which self-hosting shares with everything else. The available executor is the grok CLI (`grok`) — headless coding agent, two models (`grok-build`, xAI's Grok 4.3; `grok-composer-2.5-fast`, Cursor's Composer — the CLI *default* is Composer, so the model must always be passed explicitly), native `--worktree` support, no effort knob. The AlphaMind dogfood (its `docs/agents/grok-cli.md`) supplies the hard-won operational record: on a head-to-head eval grok-build matched or beat sonnet on pure mechanical work and was faster wall-clock, but **over-deleted behavioral tests** the moment judgment was required; it **self-reports success even when truncated**; multi-file edits exhaust low `--max-turns` and stop **without committing**; ≥3 concurrent jobs trip the team rate limit; `grok models` misreports auth. Two structural constraints: the workflow script can only spawn Claude agents (no shell, and the Agent surface speaks only Claude models), and the self-hosting checkout is load-bearing — an `--always-approve` agent must not edit it directly. Resolved by grilling, 2026-07-03.

**Decision.**

- **A dedicated driver agent; the CLI never replaces an agent.** When a task's tier is `rote` and the resolved binding says `via: grok-cli`, the workflow spawns **`drive`** (`agents/drive.md`, a Claude agent, its own model bound by the `drive` role — sonnet by default: it must tell truncation from real defect and spot over-deletion). A Claude verifier is mandatory regardless of executor, because the executor's self-report is untrusted by the eval record. The branch-protocol / booking / crash-healing prose the driver shares with build **extracts from `agents/build.md` into one protocol document both agents reference** — no duplication of the plugin's most battle-tested prose, no divergence. (Placement is settled at Plan: whether the harness scans `agents/` subdirectories as definitions is unverified — a plan-time probe.)
- **Drive choreography.** Assemble the prompt file from the task-contract slice plus the build constitution and the task's selected standards (the same craft baseline a build agent carries — the CLI inherits project conventions but the contract rides the prompt, never an assumed side channel). Run headless in a **grok-native worktree** cut at the feature-branch tip: `--worktree --worktree-ref loop/<feature-id> --always-approve --no-subagents --max-turns 500 --output-format plain`, model passed explicitly, no effort flag (unsupported). **Verify inside the worktree before anything folds back:** per-criterion tests present and green, lint clean, diff reviewed for unintended files and deleted behavioral tests, footprint against the contract, and the commit actually exists. Then fold the commit onto `loop/<feature-id>` (the standard `<feature-id>/<task-id>: <title>` message, so crash-healing's commit-pattern search still works), dispose the worktree, and book the completion report exactly as a build agent would (ADR-0029 self-booking).
- **Failure typing.** The launch leg **pre-flights** the CLI whenever an in-scope binding says `via: grok-cli` — `which grok` plus an auth smoke test (`grok -p "say PONG" --max-turns 1`; `grok models` lies about auth) — so the environment problem surfaces before a run burns budget. A drive-time CLI/auth/rate-limit failure is **environment-shaped** (the run halts, ADR-0029 typing — a tooling precondition down is the same species as the probe being down). **Truncation** (uncommitted worktree, verification never reached) is distinguished from defect and gets **one retry in a fresh worktree**, then a **feature-shaped park** whose menu offers rebind-this-task-to-a-Claude-tier or re-spec. v1's fully sequential runs satisfy the ≤2-concurrent rate-limit constraint for free — recorded as a real constraint for `worktree-parallelism`.
- **Off by default; the binding is the switch.** The shipped table binds `build.rote` to sonnet. A project opts in with one override: `"build.rote": { "model": "grok-build", "via": "grok-cli" }`. No separate delegation flag exists.
- **A port.** `delegate-executor` joins the ports inventory (optional tier): any headless autonomous coding CLI honoring the contract — prompt-file in, committed edit in an isolated worktree out, machine-checkable completion — swaps in behind the same driver.

**Why.** The driver architecture puts the untrusted, cheap, fast thing inside a verified envelope instead of pretending it's a peer: grok does the typing, Claude owns the judgment that the eval showed grok lacks, and the booking path — the part that must never lie — never leaves Claude. Extraction-not-duplication keeps one copy of the branch protocol that three prior features hardened. The native worktree quarantines blanket tool-approval away from the load-bearing checkout, and makes a truncated run's debris disposable instead of a dirty tree the next spawn's clean-tree gate trips over.

**Considered and rejected.** Grok replaces the agent outright (structurally unavailable — the workflow spawns only Claude agents and the script has no shell — and the booking would rest on a self-report the record shows is unreliable); delegation inside the build agent (a delegation path woven into `build.md` bloats the densest prose in the plugin and can't be validated independently; the dedicated driver + shared protocol doc buys the same reuse without the entanglement); editing directly on the feature branch in the main checkout (an `--always-approve` non-Claude agent inside the load-bearing self-hosted repo; truncation leaves a dirty tree exactly where the clean-tree gate will halt the next run); role-level `build → grok` binding without tiers (judgment tasks leak to the executor that over-deletes behavioral tests — the exact AlphaMind failure); the build agent deciding at runtime to delegate (invisible to the binding table, unbounded cost visibility, and the routing never lands in a durable artifact); trusting grok's own completion report with spot checks (the eval's one unambiguous lesson is that the report and the work diverge precisely when it matters).
