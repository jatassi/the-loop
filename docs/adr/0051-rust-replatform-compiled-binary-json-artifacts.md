# ADR-0051 Â· The deterministic toolchain replatforms to a compiled Rust binary, and durable artifacts move to tool-owned JSON

**Context.** The CLI is Node ESM (`plugin/bin/the-loop.js` + `plugin/src/*.js`, one vendored dependency `yaml`), invoked by skills and agents via Bash with JSON on stdout and exit codes as gates. It rests on an undeclared, unenforced assumption â€” a JS runtime on the user's machine â€” that is provably false for a meaningful slice of users: Claude Code's native installer ships a Bun-compiled binary and never places `node` on PATH, and no harness doc asserts a runtime floor beyond a POSIX shell + ripgrep (survey in `docs/briefs/rust-cli-replatform.md`, sources checked 2026-07-08). ADR-0048 vendored `yaml` to make the bundle self-contained and named eliminating the runtime dependency "the cleanest end-state and a legitimate later simplification". Separately, the tool's core operation today is comment/format-preserving YAML round-trip mutation â€” the one workload Rust does badly (`serde_yaml` is lossy and archived; no CST-preserving equivalent of JS's `yaml` exists).

**Decision.**

- **The CLI is rewritten in Rust** â€” a crate at `cli/` under a workspace root `Cargo.toml`, binary named `the-loop` â€” at strict behavior parity over the **full current command surface** (`status`, `list`, `check`, `set-status`, `plan parse|check|task`, `prepare-execution-context` incl. `--script-out`, `worktree-create|remove`, `executors-list`, `models-list`, `hooks-list`, `hooks-set`, `calibration-summarize`). The invocation seam is untouched: CLI via Bash, JSON stdout, exit-code gates. Parity binds stdout JSON and exit codes; stderr wording is informational and may normalize.
- **Durable payloads become pure tool-owned JSON files**: `docs/feature-graph.json`, `docs/plans/<id>/plan.json`, `docs/calibration/runs/<stamp>.json`. Executor playbooks stay prose markdown; only their machine-block fence flips `yaml` â†’ `json`. The tool re-emits canonically (schema key order, 2-space indent, trailing newline): hand-edited *content* survives, incidental *formatting* normalizes. YAML comments become schema fields (`section` for milestone grouping, `notes`). The round-trip-preservation machinery is deleted, not ported. Clean break: no YAML read path survives.
- **Distribution is beads-literal**: `cargo-dist` publishes checksummed per-target archives plus generated shell/powershell installers to GitHub Releases for five targets (aarch64/x86_64-apple-darwin, x86_64/aarch64-unknown-linux-musl, x86_64-pc-windows-msvc). The human installs once via the `curl | sh` one-liner; every loop surface invokes bare `the-loop` on PATH; a missing binary fails with the install one-liner as its stated remedy. No compiled artifact is ever committed to the repo.
- **The cutover is the atomic swap**: the full Rust CLI is built to parity side-by-side behind a dual-driver black-box oracle (subprocess-only, paired YAML/JSON fixtures, run against both binaries), then one human-gated session landing (`json-cutover`) migrates this repo's artifacts, flips every invocation site, and deletes the JS path. The flip deliberately does **not** run through the execution pipeline: it swaps the tool the pipeline's own post-merge machinery (set-status, the record agent's `calibration-summarize`) runs on, so landing it mid-run would strand that machinery against a deleted CLI. Self-edits take effect on the next run â€” the ADR-0034 posture.
- **The Rust quality profile carries the aggressive-linting regime**: workspace lints deny warnings with clippy's `all`, `pedantic`, `nursery`, and `cargo` groups enabled, `allow` attributes require reasons, and `cargo fmt --check` rides the gate.

**Why.** Zero runtime dependency on the user's machine is the defining constraint, and a compiled binary is the only shape that discharges it rather than declaring it. Rust fits once the format decision removes its one real disadvantage: the workload is serde-driven parse/transform/print with little concurrency or lifetime complexity, and compile-time correctness aligns with the project's ethos. JSON + canonical emit converts the tool's hardest part (lossless round-trip) into its most trivial. PATH distribution was chosen at the design interview (2026-07-09) over plugin-owned auto-fetch: one manual install step buys freedom from session-start network dependencies and a resolver shim to maintain, and it is the exact UX `beads` has already proven for this consumer.

**Considered and rejected.** *Accept-and-declare the node floor* â€” fails native-installer users outright; the assumption is the defect. *MCP* â€” adds a server-runtime problem while solving nothing for deterministic local logic. *`bun build --compile`* â€” embeds the ~100MB Bun runtime per platform, forcing the fetch model at far worse size. *Go* â€” the lower-risk port (trivial cross-compile, healthy YAML libs) but YAML support stopped mattering once the format moved, and Rust won on alignment and maintainer motivation. *Committing prebuilt binaries* â€” per-arch blobs bloat the bundle, Gatekeeper quarantines unsigned downloads, blobs are unreviewable, and the running artifact diverges from the source of record. *A shell-shim resolver or SessionStart auto-fetch* â€” keeps "installing the plugin fetches the binary" but adds network at session start and a second distribution mechanism to maintain; declined at the interview in favor of PATH. *Markdown wrappers with fenced JSON blocks* â€” ports the fenced-block extraction machinery the move exists to delete and keeps the tool owning only part of each file. *A generated `feature-graph.md` view* â€” two files that agree only by discipline, derived-state churn in every diff.
