# ADR-0026 · Git strategy: feature branches, a configurable integration target, validate-then-squash-merge

**Context.** Build and Validate were blocked on the branching/integration strategy (feature-graph note, 2026-07-01 review): where a validated feature integrates, a parked feature's branch lifecycle across run boundaries, who rebases a rotted parked branch on fix-in-place, and crash-recovery commit granularity. Resolved by grilling, 2026-07-01.

**Decision.**
- **One feature branch per feature** — `loop/<feature-id>`, cut from the integration target when Build picks the feature up. v1's sequential tasks commit serially onto it; when [[worktree-parallelism]] lands, per-task branches fork off and merge back into *this same branch* — the integration spine never changes shape.
- **One commit per task.** A build agent commits exactly once per completed task, message stamped `<feature-id>/<task-id>`; the [[completion report]]'s `footprint_actual`/`diff_actual` derive from that commit. **The task is the crash-recovery quantum:** an interrupted task's uncommitted work is discarded, the branch tip is the last completed task, and the next stateless run re-runs the interrupted task fresh — a half-done task is not resumable without the dead agent's context.
- **The [[integration target]] is a bound ref, default `main`.** One knob, bound at Design (already human-gated), recorded in the design artifact so every stateless run reads the same answer; the run machinery is target-agnostic. Default posture is trunk-based: validated features merge to `main` as the run goes — `main` means *everything validated so far*, and Ship gates **deployment, not integration**. A rewrite-scale intake binds an intake branch (`loop/intake/<name>`) instead; Ship merges it to `main` when the intake ships.
- **Validate, then squash-merge — the target only ever receives validated work.** The [[independent validator]] readies the branch (rebasing onto the current target tip if it moved), runs its legs on the branch's tree, and only on a perfect verdict squash-merges: **one commit per validated feature** on the target (message: feature id + `design_version`). Reverting a feature — including a bad self-edit under self-hosting — is one clean revert. Task granularity is not lost; it lives in the [[plan artifact]] and completion reports. This is Build's acceptance criterion made literal: a feature's tasks produce a single merged diff.
- **Branches carry code only; bookkeeping commits on the target at phase boundaries.** The plan artifact commits at Plan; at validate-or-park, one boundary step folds completion reports into the plan, flips the feature's graph status, and re-renders the Ledger. The target is always the complete state machine — a stateless run reads it and knows everything; parked state never hides on an unmerged branch.
- **Branch lifecycle invariant: a `loop/<feature-id>` branch exists iff its feature is building or parked.** A clean squash-merge deletes the branch immediately. A parked feature's [[escalation record]] names the branch ref; a resolution that discards the work (re-plan, re-slice, abandon) deletes the branch in the same fold-in step. The branch inventory always mirrors the graph's in-flight set — no GC pass, nothing lingers.
- **Rot is handled at resume, by the resuming build agent.** Fix-in-place starts with a rebase of the parked branch onto the current target — rebase, not merge-in: branches are loop-private so rewrite is safe, the one-commit-per-task shape survives, and completion reports reference paths, not SHAs. A rebase conflict re-parks the feature as a conflict-shaped [[deviation]] with its own menu.
- **Clean-tree precondition.** A run launches only from a clean tree at the integration-target checkout. A dirty tree is surfaced neutrally as a blocking condition — it is *not attributed* (crash leftovers and human work-in-progress look identical); the loop never stashes, resets, or cleans on its own.

**Consequences.** All loop refs live under the `loop/` namespace (`loop/<feature-id>`, future `loop/<feature-id>/<task-id>` task branches, `loop/intake/<name>`). v1 is local-only — pushing is a Ship/human concern, not the run's. This amends ADR-0012's merge-fold-in: integration is still the validator's setup step (assembly, not authoring), but ordered validate-then-merge with a squash, and when worktree-parallelism lands, task branches merge into the *feature branch*, never directly into the target.

**Considered and rejected.** Per-task branches in v1 (sequential tasks never race — moving parts with zero benefit); committing straight to the target (invalid work lands on `main`, demanding revert machinery — fails loudly *and* unsafely); a per-run integration branch as the default (a second durable ref whose meaning shifts per run; the intake-scoped binding covers the real need); merge-commits preserving task history on the target (feature-revert becomes `-m 1` archaeology; task noise interleaves across features); bookkeeping riding the feature branch (a parked feature's state never reaches the target — breaks the stateless pass); auto-stash on a dirty tree (the loop mutating human work-in-progress; a stash-pop conflict after a long run destroys trust); merge-target-into-branch freshening (history noise; breaks per-task commit shape); a run-start refresh sweep over parked branches (spend on features the human may discard); merging before validating (unvalidated work on the target guts the [[perfection bar]]).
