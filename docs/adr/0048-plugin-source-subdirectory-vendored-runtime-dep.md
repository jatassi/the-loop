# ADR-0048 · Plugin content lives in a source subdirectory, and the one runtime dependency is vendored

**Context.** The repo root is the plugin root (`.claude-plugin/marketplace.json` binds `source: "./"`). Claude Code installation copies the plugin root **wholesale** — there is no `.gitignore` respect, no `files` allowlist, no `.npmignore` equivalent (docs checked 2026-07-08). So the bundle carries `eval/` (megabytes), `docs/`, `test/`, and every devDependency. The v0.4.6 release aborted at the gate on this. Two facts constrain the fix: (1) `plugin.json`'s custom-path fields steer discovery, not what is copied — only a physical directory boundary can scope the bundle; (2) installed plugins **cannot reference files outside the plugin root** (`../` paths are dropped at install), so the CLI's one npm runtime dependency (`yaml`) must ship *inside* the bundle, yet `node_modules/` is gitignored and today reaches the bundle only by the accident of local-path installs copying the working tree.

**Decision.**
- **Plugin content moves under `plugin/`.** The plugin root becomes `<repo>/plugin/` with `plugin/.claude-plugin/plugin.json`; `.claude-plugin/marketplace.json` stays at the repo root with `source: "./plugin"`. Dev tooling — `docs/`, `test/`, `eval/`, the root `package.json`/`eslint.config.js`, and the dev `node_modules/` — stays at the repo root, outside the bundle. `PLUGIN_ROOT` (derived from `import.meta.url`) and braced `${CLAUDE_PLUGIN_ROOT}` both self-rehome, so the move is almost entirely mechanical.
- **The `yaml` dependency (v2.9.0, zero transitive deps) is vendored** as tracked content at `plugin/node_modules/yaml/`, so `import 'yaml'` resolves unchanged and it ships under a git-source clone (which carries only tracked files). A `plugin/package.json` declaring `"type": "module"` ships with it — the installed bundle is *only* `plugin/`, so without it Node would treat the installed ESM as CommonJS.

**Why.** The directory boundary is the only mechanism the harness offers for scoping a bundle, and the sanctioned marketplace-source-subdirectory pattern is exactly that. Vendoring is the smallest, safest way to make an underivable runtime input present: it preserves the project's "plain ESM JS / no build" posture, needs no network or session-start machinery, and survives the shift from local-path installs to a published git source. The cost — ~1.2M of tracked third-party source and one un-ignored `node_modules` path — is bounded and honest; `yaml` has zero transitive deps, so the vendored tree is a single package.

**Considered and rejected.** A `files`/allowlist manifest field — does not exist. Keeping the flat layout and excluding dev content via discovery paths — discovery is not copy-scope; the whole root still ships. A **SessionStart hook** that `npm install`s `yaml` into `${CLAUDE_PLUGIN_DATA}` with `NODE_PATH` plumbing (the documented deps pattern) — adds a hook, a session-start network dependency, and `NODE_PATH` wiring that does not fit the `node "${CLAUDE_PLUGIN_ROOT}/bin/the-loop.js"` Bash invocations agents use; most machinery for the least benefit. **Eliminating `yaml`** (hand-rolled parse/emit → zero runtime deps → nothing to bundle) — the cleanest end-state and a legitimate later simplification, but it risks the graph's byte-round-trip guarantee and is far larger than a release-blocker should carry. A **build/bundler step** inlining `yaml` — violates the no-build posture.
