# ADR-0031 · Delegated executors: an executor registry of CLI playbooks, driven by a verifying Claude driver

**Context.** ADR-0030's `via` field needs an execution protocol. The motive is budget: rote work off the Claude usage budget, which self-hosting shares with everything else. More than one executor CLI is expected — the grok CLI today (`grok-build`, xAI's Grok 4.3; `grok-composer-2.5-fast`, Cursor's Composer — the CLI *default* is Composer, so the model must always be passed explicitly), codex- or gemini-style headless CLIs plausibly next — so the protocol must not bake any single CLI into the binding shape, the driver, or the pre-flight. The AlphaMind dogfood (its `docs/agents/grok-cli.md`) supplies the operational record for the first executor: on a head-to-head eval grok-build matched or beat sonnet on pure mechanical work and was faster wall-clock, but **over-deleted behavioral tests** the moment judgment was required; it **self-reports success even when truncated**; multi-file edits exhaust low `--max-turns` and stop **without committing**; ≥3 concurrent jobs trip the team rate limit; `grok models` misreports auth. Two structural constraints: the workflow script can only spawn Claude agents (no shell, and the Agent surface speaks only Claude models), and the self-hosting checkout is load-bearing — a blanket-approval agent must not edit it directly. Resolved by grilling, 2026-07-03; generalized from one CLI to a registry the same day.

**Decision.**

- **Executors register by playbook.** One file per executor: `executors/<id>.md` in the plugin — a YAML **machine block** (id, command, model ids, headless invocation template, `worktree: native | driver-made`, availability check, auth smoke test) under **narrative operational lore** (truncation signatures, rate limits, gotchas — knowledge a Claude driver can actually use, which pure JSON would flatten into strings). Registration is the file's presence. The resolver validates a binding's `via` against the registry and its `model` against the playbook's list; an unregistered `via` is a resolution error surfaced at pre-flight, never silently. v1 ships the grok playbook only; codex- or gemini-style CLIs register the same way; **project-local playbook dirs defer to `ports-adapters-full`** — adapter extension is that feature's job.
- **One dedicated driver agent; a CLI never replaces an agent.** When a task's tier is `rote` and the resolved binding names an executor, the workflow spawns **`drive`** (`agents/drive.md`, a Claude agent serving *all* executors, parameterized by the playbook; its own model bound by the `drive` role — sonnet by default: it must tell truncation from real defect and spot over-deletion; per-executor driver models remain available for free as dotted sub-roles, `drive.<executor>`, through ADR-0030's open registry). A Claude verifier is mandatory regardless of executor, because CLI self-reports are untrusted by the eval record. The branch-protocol / booking / crash-healing prose the driver shares with build **extracts from `agents/build.md` into one protocol document both agents reference** — no duplication of the plugin's most battle-tested prose, no divergence. (Placement is settled at Plan: whether the harness scans `agents/` subdirectories as definitions is unverified — a plan-time probe.)
- **Drive choreography, playbook-parameterized.** Assemble the prompt file from the task-contract slice plus the build constitution and the task's selected standards (the same craft baseline a build agent carries — the contract rides the prompt, never an assumed side channel). Run the CLI headless per the playbook's invocation template in a **worktree cut at the feature-branch tip** — native when the playbook says so (grok: `--worktree --worktree-ref loop/<feature-id> --always-approve --no-subagents --max-turns 500 --output-format plain`, no effort flag), otherwise the driver makes one. **Verify inside the worktree before anything folds back:** per-criterion tests present and green, lint clean, diff reviewed for unintended files and deleted behavioral tests, footprint against the contract, and the commit actually exists. Then fold the commit onto `loop/<feature-id>` (the standard `<feature-id>/<task-id>: <title>` message, so crash-healing's commit-pattern search still works), dispose the worktree, and book the completion report exactly as a build agent would (ADR-0029 self-booking).
- **Failure typing.** The launch leg **pre-flights every distinct in-scope executor per its playbook** — availability check plus auth smoke test (grok's: `grok -p "say PONG" --max-turns 1`; `grok models` lies about auth) — so environment problems surface before a run burns budget. A drive-time CLI/auth/rate-limit failure is **environment-shaped** (the run halts, ADR-0029 typing — a tooling precondition down is the same species as the probe being down). **Truncation** (uncommitted worktree, verification never reached) is distinguished from defect and gets **one retry in a fresh worktree**, then a **feature-shaped park** whose menu offers rebind-this-task-to-a-Claude-tier or re-spec. v1's fully sequential runs satisfy per-executor concurrency limits for free (grok's is ≤2) — recorded as a real constraint for `worktree-parallelism`.
- **Off by default; the binding is the switch.** The shipped table binds `build.rote` to sonnet. A project opts in with one override: `"build.rote": { "model": "grok-build", "via": "grok" }`. No separate delegation flag exists.
- **A multi-adapter port.** `delegate-executor` joins the ports inventory (optional tier) with a deliberate extension to port semantics: its adapters — the playbooks — **bind concurrently, keyed by executor id**, and the bindings' `via` field selects among them per role and task. Any headless autonomous coding CLI honoring the contract — prompt-file in, committed edit in an isolated worktree out, machine-checkable completion — registers behind the same driver.

**Why.** The driver architecture puts the untrusted, cheap, fast thing inside a verified envelope instead of pretending it's a peer: the executor does the typing, Claude owns the judgment the eval showed executors lack, and the booking path — the part that must never lie — never leaves Claude. The playbook registry keeps the protocol executor-agnostic where it must be (binding shape, driver, pre-flight, failure typing) and executor-specific exactly where reality is (invocation, worktree support, failure signatures) — adding a CLI is adding a file, not amending an ADR. Extraction-not-duplication keeps one copy of the branch protocol that three prior features hardened. The worktree quarantines blanket tool-approval away from the load-bearing checkout, and makes a truncated run's debris disposable instead of a dirty tree the next spawn's clean-tree gate trips over.

**Considered and rejected.** A CLI replacing the agent outright (structurally unavailable — the workflow spawns only Claude agents and the script has no shell — and the booking would rest on a self-report the record shows is unreliable); delegation inside the build agent (a delegation path woven into `build.md` bloats the densest prose in the plugin and can't be validated independently; the dedicated driver + shared protocol doc buys the same reuse without the entanglement); a single hardwired `via: grok-cli` (bakes one vendor into the binding shape, the driver, and the pre-flight — the registry costs one indirection and un-bakes all three); pure-JSON executor profiles (the operational lore the driver needs — how truncation manifests, which auth checks lie — is prose pretending to be data); prose-only playbooks (the resolver and pre-flight lose mechanical model validation and smoke tests — the visible-fallback and pre-flight guarantees go soft); editing directly on the feature branch in the main checkout (a blanket-approval non-Claude agent inside the load-bearing self-hosted repo; truncation leaves a dirty tree exactly where the clean-tree gate will halt the next run); role-level `build → executor` binding without tiers (judgment tasks leak to executors that over-delete behavioral tests — the exact AlphaMind failure); the build agent deciding at runtime to delegate (invisible to the binding table, unbounded cost visibility, and the routing never lands in a durable artifact); trusting the executor's own completion report with spot checks (the eval's one unambiguous lesson is that the report and the work diverge precisely when it matters).
